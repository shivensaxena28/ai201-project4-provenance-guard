# Provenance Guard — Planning & Spec

A backend service that any creative-sharing platform can plug in to classify
whether submitted text was written by a human or generated by AI, score how
*confident* that classification is, surface a plain-language transparency label
to readers, and let creators appeal a decision they think is wrong.

This document is the spec the implementation was written against. It is
deliberately concrete: every threshold, label string, and signal output shape
below maps directly to code in `detection.py`, `labels.py`, `storage.py`, and
`app.py`.

---

## 1. Architecture narrative

A single piece of text flows through the system like this:

1. A creator (or the host platform on their behalf) sends `POST /submit` with
   `{ "text": ..., "creator_id": ... }`.
2. **Rate limiting** (Flask-Limiter) checks the caller hasn't exceeded
   10 submissions/minute or 100/day. If they have, the request is rejected with
   `429` before any work is done.
3. The text is run through the **detection pipeline** (`detection.classify`):
   - **Signal 1 — LLM classification (Groq):** the text is sent to
     `llama-3.3-70b-versatile`, which returns an `ai_probability` in `[0,1]`.
   - **Signal 2 — Stylometric heuristics:** pure-Python structural metrics
     (sentence-length burstiness, long-word/formality ratio, transition-marker
     density, informality markers) are folded into an `ai_likelihood` in `[0,1]`.
4. **Confidence scoring** combines the two into a single `ai_score`
   (`0.65·llm + 0.35·stylo`), derives a verdict via asymmetric thresholds, and
   computes a `confidence` that is eroded when the two signals disagree.
5. **Transparency label** (`labels.make_label`) maps the verdict + confidence to
   one of three reader-facing label variants.
6. **Audit log** (`storage.record_classification`) writes a structured row
   capturing the content id, creator, timestamp, verdict, combined confidence,
   both individual signal scores, and the full signal breakdown.
7. The endpoint returns JSON: `content_id`, `attribution`, `confidence`,
   `ai_score`, the per-signal scores, the `label`, and `status`.

The **appeal flow** is shorter: `POST /appeal` with a `content_id` and
`creator_reasoning` looks up the original decision, flips the content's status
to `under_review`, writes a second audit-log row (event `appeal`) carrying the
creator's reasoning *next to* the original scores, and returns a confirmation.

### The false-positive scenario (this drove the design)

Worst case on a *writing* platform: a real human writer — say a non-native
English speaker whose prose is formal and even — gets labeled "AI-generated."
That's a public accusation against their authorship. We trace it through:

- Stylometry will (correctly, structurally) score formal/even prose as fairly
  AI-like. The LLM may *also* over-call formal academic prose as AI.
- So the combined `ai_score` can land high (~0.74 in testing).
- **Mitigation 1 — asymmetric thresholds:** the bar to call something
  `likely_ai` is `0.75`, much higher than the `0.35` bar for `likely_human`.
  The wide `0.35–0.75` "uncertain" band is intentionally tilted away from AI
  accusations, so borderline formal prose lands in *uncertain*, not *likely_ai*.
- **Mitigation 2 — the label never claims certainty:** even `likely_ai` says
  "estimate, not a certainty" and points to the appeal path.
- **Mitigation 3 — appeals:** the creator files an appeal; status becomes
  `under_review` and a human takes over. No automated re-classification — a
  human is the backstop for exactly this failure mode.

## 2. The five required spec questions

### 2.1 Detection signals

| | Signal 1 — LLM classification | Signal 2 — Stylometric heuristics |
|---|---|---|
| **Tool** | Groq `llama-3.3-70b-versatile` | Pure Python (`detection.py`) |
| **Measures** | Holistic *semantic / stylistic coherence* — does the text read as generic, even-toned, "essay-like," hedged (AI) vs. idiosyncratic, concrete, messy (human)? | *Structural* statistics: sentence-length variance, long-word ratio, transition-marker density, informality markers. |
| **Why it differs human↔AI** | Models produce fluent, coherent, but often generic prose lacking lived specificity. The LLM judges this the way a careful reader would. | AI text is metronomic and formal; human prose is bursty and informal. These are measurable without any model. |
| **Output shape** | `ai_probability` ∈ `[0,1]` (1 = almost certainly AI) + one-sentence reasoning. | `ai_likelihood` ∈ `[0,1]` (1 = most AI-like) + a dict of interpretable sub-scores. |
| **Blind spot** | Lightly human-edited AI; over-calls formal/non-native human writing; opaque black-box with its own biases. | Short texts (too few sentences for variance to matter); deliberately formal human writing scores AI-like. |

These two are *genuinely independent*: one is semantic, one is structural. When
they **agree**, that is real corroboration; when they **disagree**, that is real
uncertainty — which is why disagreement directly lowers the reported confidence.

**Combination** (`detection.classify`):
```
ai_score   = 0.65 · llm_score + 0.35 · stylo_score
disagree   = |llm_score − stylo_score|
directional = max(ai_score, 1 − ai_score)
confidence = directional · (1 − 0.25·disagree)     # ×0.85 if LLM unavailable
```
The LLM is weighted higher (0.65) because it is the more reliable holistic
judge; stylometry (0.35) corroborates and catches the model's blind spots.

### 2.2 Uncertainty representation

We decided what the *number means to a reader* first, then built toward it:

- **`confidence` is the system's certainty in the verdict it is stating**, on a
  0–1 scale shown to users as a percentage. ~0.50 means a coin-flip; ~0.95 means
  near-certain.
- A `0.51` and a `0.95` land in different label variants *and* show different
  percentages — there is no binary flip at 0.5.
- Raw signal outputs map to a calibrated score via the weighted blend above,
  with confidence *reduced* when the two independent signals disagree, so
  "both signals strongly agree" reads as more confident than "the average
  happened to land here."

**Thresholds on `ai_score`** (verdict bands):

| `ai_score` | Verdict | Label variant |
|---|---|---|
| `≥ 0.75` | `likely_ai` | high-confidence AI |
| `0.35 – 0.75` | `uncertain` | uncertain |
| `≤ 0.35` | `likely_human` | high-confidence human |
| (signals disagree by `> 0.45`) | forced `uncertain` | uncertain |

The bands are **asymmetric on purpose**: the uncertain zone is centered at
`0.55`, not `0.50`, so it takes stronger evidence to accuse a human's work of
being AI than to clear it as human.

### 2.3 Transparency label design (three variants, exact text)

The `{pct}` placeholder is the rounded `confidence` as a percentage.

**High-confidence AI** (`likely_ai`):
> 🤖 **Likely AI-generated**
> Our automated analysis estimates this content was probably created with the help of generative AI (confidence: {pct}). This is an estimate, not a certainty — AI detection is an unsolved problem. If you are the creator and believe this is wrong, you can appeal and a human will review it.

**High-confidence human** (`likely_human`):
> ✍️ **Likely human-written**
> Our automated analysis found no strong signs of AI generation in this content (confidence: {pct}). This is an estimate, not a guarantee of authorship.

**Uncertain** (`uncertain`):
> ❓ **Attribution uncertain**
> Our automated analysis could not confidently determine whether this content was written by a human or generated by AI (confidence in either direction: {pct}). Rather than guess, we are telling you honestly: treat the origin as unverified.

### 2.4 Appeals workflow

- **Who:** the creator (or platform on their behalf) who owns the `content_id`.
- **What they provide:** the `content_id` and free-text `creator_reasoning`.
- **What the system does** (`storage.record_appeal`):
  1. Look up the original decision by `content_id` (404 if unknown).
  2. Update that content's `status` → `under_review`.
  3. Append an audit-log row (event `appeal`) carrying the creator's reasoning
     *and* the original verdict/scores, so a reviewer sees both side by side.
  4. Return a confirmation echoing the original attribution and the new status.
- **No automated re-classification** — a human is the decision-maker. The appeal
  just routes the case to them.
- **What a reviewer sees in the queue:** `GET /log` filtered to `event=appeal`
  rows shows, per case: the content id, creator id, original attribution +
  confidence, both signal scores, the creator's reasoning, and timestamps.

### 2.5 Anticipated edge cases (specific, not generic)

1. **Formal / non-native human writing** (e.g. an academic paragraph on monetary
   policy). High long-word ratio, even sentence lengths, and transition words
   make stylometry score it AI-like, and the LLM tends to over-call it too. Our
   testing showed `ai_score ≈ 0.74` for exactly this. Mitigation: the high
   `0.75` AI threshold keeps it in *uncertain* rather than a false AI label, and
   the appeal path exists for it.
2. **A short, repetitive, simple-vocabulary poem.** Heavy repetition deflates
   lexical variety and uniform short lines flatten sentence-length variance, so
   stylometry may read it as "uniform → AI" even though it is a deliberate human
   craft choice. With few sentences, stylometry is least reliable here.
3. **Lightly human-edited AI output.** A human pass adds enough irregularity to
   fool both signals toward the middle — by design this lands in *uncertain*
   (tested at `ai_score ≈ 0.65`), which is the honest answer.

## Architecture

```
                      ┌─────────────────────────────────────────────────────────┐
                      │                  PROVENANCE GUARD API                    │
                      └─────────────────────────────────────────────────────────┘

SUBMISSION FLOW
===============

  creator
    │  POST /submit { text, creator_id }
    ▼
 ┌──────────────┐   request allowed    ┌───────────────────────────────────────┐
 │ Rate limiter │ ───────────────────► │           detection.classify          │
 │ 10/min;100/d │                      │                                        │
 └──────┬───────┘                      │   raw text          raw text           │
        │ over limit                   │      │                  │              │
        ▼                              │      ▼                  ▼              │
     429 JSON                          │ ┌──────────┐      ┌──────────────┐     │
                                       │ │ Signal 1 │      │  Signal 2    │     │
                                       │ │   LLM    │      │ stylometry   │     │
                                       │ │ (Groq)   │      │ (pure Python)│     │
                                       │ └────┬─────┘      └──────┬───────┘     │
                                       │ llm_score 0..1   stylo_score 0..1      │
                                       │      └────────┬─────────┘              │
                                       │               ▼                        │
                                       │   ai_score = 0.65·llm + 0.35·stylo     │
                                       │   confidence = f(ai_score, disagree)   │
                                       └───────────────┬────────────────────────┘
                                                       │ verdict + confidence
                                                       ▼
                                            ┌─────────────────────┐
                                            │  labels.make_label  │  label text
                                            └──────────┬──────────┘
                                                       │ full record
                          ┌────────────────────────────┼───────────────────────┐
                          ▼                                                     ▼
                 ┌──────────────────┐  content_id, scores,        ┌──────────────────────┐
                 │  audit_log (DB)  │  attribution, status        │  JSON response to     │
                 │  event=classified│ ◄──────────────             │  caller (incl. label) │
                 └──────────────────┘                             └──────────────────────┘


APPEAL FLOW
===========

  creator
    │  POST /appeal { content_id, creator_reasoning }
    ▼
 ┌───────────────────────┐   lookup by content_id   ┌─────────────────────┐
 │  storage.record_appeal│ ───────────────────────► │  content row        │
 │                       │   status → under_review  │  (original decision)│
 └───────────┬───────────┘                          └─────────────────────┘
             │ append audit row (event=appeal, carries reasoning + original scores)
             ▼
   ┌──────────────────┐                         ┌────────────────────────────┐
   │  audit_log (DB)  │                         │  JSON confirmation:          │
   │  event=appeal    │  ─────────────────────► │  status=under_review,        │
   │  status=under_   │                         │  original attribution echoed │
   │  review          │                         └────────────────────────────┘
   └──────────────────┘
```

**Narrative.** *Submission:* raw text passes the rate limiter, fans out to the
LLM and stylometric signals in parallel concept, is combined into a single
`ai_score`/`confidence`, mapped to a transparency label, written to the audit
log, and returned. *Appeal:* a `content_id` + reasoning flips the content's
status to `under_review`, appends an appeal row to the audit log beside the
original decision, and returns a confirmation — a human reviewer takes it from
there.

## AI Tool Plan

How AI assistance was directed across the three implementation milestones, what
each was asked to produce, and how output was verified.

### M3 — Submission endpoint + first signal
- **Spec sections provided:** §2.1 Detection signals (Signal 1 row) + the
  Architecture diagram.
- **Asked to generate:** the Flask app skeleton with a `POST /submit` route stub
  and the `llm_signal(text)` function returning a structured `(score, info)`.
- **Verification:** call `llm_signal` directly on the clearly-AI and
  clearly-human samples and confirm the score moves the right direction and the
  signature matches the spec (a float in `[0,1]`, not a binary flag) *before*
  wiring it into the endpoint. Confirm `/submit` returns a `content_id`.

### M4 — Second signal + confidence scoring
- **Spec sections provided:** §2.1 Detection signals (Signal 2 row) + §2.2
  Uncertainty representation + the diagram.
- **Asked to generate:** the `stylometric_signal(text)` function (burstiness,
  long-word ratio, transition density, informality) and the `classify` combiner.
- **Verification:** run `test_pipeline.py` on all four spec inputs and check the
  generated thresholds match §2.2 exactly (AI tools sometimes invent their own
  ranges). Confirm clearly-AI ≠ clearly-human scores and all three label
  categories are reachable. Print both signal scores separately to find which
  one misbehaves if a case looks wrong.

### M5 — Production layer
- **Spec sections provided:** §2.3 Label variants + §2.4 Appeals workflow + the
  diagram.
- **Asked to generate:** `labels.make_label` (verdict + confidence → label text)
  and the `POST /appeal` endpoint.
- **Verification:** ask the label function to emit all three variants and
  diff the text against §2.3 verbatim. For appeals, confirm status flips to
  `under_review` and the appeal row appears in `GET /log` with
  `appeal_reasoning` populated. Confirm Flask-Limiter returns `429` past 10/min.
