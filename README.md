# Provenance Guard

A backend service that any creative-sharing platform can plug in to answer one
question about a piece of submitted text: **was this written by a human, or
generated by AI?** — and, crucially, *how confident are we?* It returns a
plain-language transparency label for readers, logs every decision, rate-limits
abuse, and lets creators appeal a classification they believe is wrong.

The design philosophy throughout: **perfect AI detection is an unsolved
problem, so the honest thing to do is communicate uncertainty and give creators
a path to appeal.** On a writing platform, falsely labeling a human's work as AI
is the worst outcome, so the system is deliberately biased away from that.

**📹 [Watch the demo walkthrough](https://www.loom.com/share/c6b5fc6384b640a5a4d71a9f993bcbc0)**

## Quick start

```bash
python -m venv .venv
source .venv/Scripts/activate          # Windows (Git Bash);  use .venv/bin/activate on Mac/Linux
pip install -r requirements.txt

# .env in repo root (already gitignored):
#   GROQ_API_KEY=your_key_here

python app.py          # serves on http://127.0.0.1:5000
```

The system **degrades gracefully without a Groq key**: the LLM signal returns a
neutral 0.5, the verdict falls back to stylometry, and confidence is reduced —
so it stays runnable, just less informative.

### Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/submit` | Classify text. Body: `{ "text": ..., "creator_id": ... }`. Rate limited. |
| `POST` | `/appeal` | Contest a classification. Body: `{ "content_id": ..., "creator_reasoning": ... }`. |
| `GET`  | `/log` | Recent structured audit-log entries (`?limit=N`). |

## Architecture

The full diagram and flow narrative live in [planning.md](planning.md#architecture).
In short: `POST /submit` → rate limiter → Signal 1 (LLM) + Signal 2
(stylometry) → confidence scoring → transparency label → audit log → JSON
response. `POST /appeal` → status `under_review` → audit log → confirmation.

Code map:

| File | Responsibility |
|---|---|
| `app.py` | Flask routes, rate limiting, request/response shaping |
| `detection.py` | Both signals + confidence combination/calibration |

## Detection signals — what and why

The pipeline uses **two genuinely independent signals**. They are independent
because one is *semantic* and one is *structural*, so their agreement is real
corroboration and their disagreement is real uncertainty.

### Signal 1 — LLM classification (Groq `llama-3.3-70b-versatile`)

Asks a large model to holistically judge whether the text reads as human- or
AI-written, returning `ai_probability ∈ [0,1]` plus a one-sentence reason.

- **Captures:** semantic and stylistic coherence — generic phrasing, even tone,
  lack of lived specificity, "essay-like" structure. The things a careful human
  reader notices that don't reduce to a formula.
- **Why I chose it:** it is the strongest single judge of the *gestalt* of a
  text, and it generalizes to writing styles a hand-coded heuristic would miss.
- **Blind spot:** it over-calls formal and non-native human writing as AI, it is
  fooled by lightly human-edited AI, and it is an opaque black box with its own
  biases. This is exactly why it is not used alone.

### Signal 2 — Stylometric heuristics (pure Python)

Measurable structural statistics combined into `ai_likelihood ∈ [0,1]`:

| Sub-metric | What it captures | AI tendency |
|---|---|---|
| **Sentence-length uniformity** (coefficient of variation) | "Burstiness" — humans vary sentence length wildly; AI is metronomic | low variance → AI |
| **Long-word / formality ratio** | Fraction of words ≥ 7 letters | high → AI/formal |
| **Transition-marker density** | "Furthermore", "It is important to note", etc. per sentence | high → AI |
| **Informality markers** (inverted) | Contractions, lowercase "i", slang, ALL-CAPS emphasis | high → human |

- **Why I chose it:** it is completely independent of the LLM, costs nothing,
  needs no network, and catches the LLM's blind spots from a different angle.
- **Blind spot:** short texts (too few sentences for variance to be meaningful)
  and deliberately formal human prose, which scores AI-like on every metric.

**Combination:** `ai_score = 0.65·llm_score + 0.35·stylo_score`. The LLM is
weighted higher because it is the more reliable holistic judge; stylometry
corroborates and backstops it.

**What I'd change deploying this for real:** I'd add a third, genuinely
different signal (e.g. token-level perplexity from a small local model, which is
independent of both the prompt-based LLM judgement and the surface statistics)
and move from a fixed weighted blend to a learned ensemble trained on a labeled
human/AI corpus. I'd also pin and version the Groq model — a silent model update
would shift every score — and cache results so re-submitting identical text
doesn't re-bill the API or produce a different verdict.

## Confidence scoring & uncertainty

I decided **what the number should mean to a reader before deciding how to
compute it.** `confidence` is the system's certainty in the verdict it is
stating, shown as a percentage: ~50% means a coin-flip, ~95% means near-certain.
A `0.51` and a `0.95` land in different label variants *and* show different
percentages — there is no binary flip at 0.5.

```
ai_score    = 0.65·llm_score + 0.35·stylo_score
disagree    = |llm_score − stylo_score|
directional = max(ai_score, 1 − ai_score)
confidence  = directional · (1 − 0.25·disagree)      # ×0.85 if LLM unavailable
```

Disagreement between the two independent signals **erodes** confidence — "both
signals strongly agree" should read as more confident than "the average happened
to land here."

**Verdict thresholds (asymmetric on purpose):**

| `ai_score` | Verdict |
|---|---|
| `≥ 0.75` | `likely_ai` |
| `0.35 – 0.75` | `uncertain` |
| `≤ 0.35` | `likely_human` |
| signals disagree by `> 0.45` | forced `uncertain` |

The uncertain band is centered at **0.55, not 0.50** — it takes stronger
evidence to accuse a human's work of being AI (`0.75`) than to clear it as human
(`0.35`). **This is where the false-positive asymmetry lives:** a borderline
formal-but-human paragraph lands in *uncertain* rather than getting a false
"AI-generated" label.

### How I tested that scores are meaningful

`test_pipeline.py` runs four deliberately chosen inputs (clearly AI, clearly
human, and two borderline cases) and prints each signal separately plus the
combined verdict. Real output (with the live Groq signal):

| Input | `llm` | `stylo` | `ai_score` | `confidence` | verdict |
|---|---|---|---|---|---|
| Clearly AI-generated (formal essay, "Furthermore…") | 0.80 | 0.727 | **0.7744** | 0.76 | `likely_ai` |
| Clearly human (casual ramen review, lowercase, slang) | 0.20 | 0.116 | **0.1707** | 0.81 | `likely_human` |
| Borderline: formal human (monetary-policy paragraph) | 0.80 | 0.622 | **0.7377** | 0.70 | `uncertain` |
| Borderline: lightly edited AI (remote-work musing) | 0.70 | 0.559 | **0.6506** | 0.63 | `uncertain` |

The clearly-AI and clearly-human cases are ~0.60 apart in `ai_score` — the
scores vary meaningfully, not a near-constant. Both borderline cases correctly
land in `uncertain`. Notably the **formal-human case (0.7377) is held back from
`likely_ai`** by the high 0.75 threshold — the false-positive protection working
as designed.

### Two example submissions with different confidence

**High-confidence case** — clearly AI text returned:
```json
{
  "attribution": "likely_ai",
  "ai_score": 0.8395,
  "confidence": 0.8032,
  "signals": { "llm_score": 0.9, "stylo_score": 0.727 }
}
```
Label shown: *"🤖 Likely AI-generated … (confidence: 80%)"*.

**Lower-confidence case** — a poetic but generic paragraph returned:
```json
{
  "attribution": "uncertain",
  "ai_score": 0.4851,
  "confidence": 0.4836,
  "signals": { "llm_score": 0.4, "stylo_score": 0.6431 }
}
```
Label shown: *"❓ Attribution uncertain … (confidence in either direction:
48%)"*. The two signals disagreed (LLM said human-ish 0.4, stylometry said
AI-ish 0.64), which is exactly the situation the system should report as
uncertain rather than guess.

**What I'd change deploying this for real:** right now `confidence` is a
*relative* measure — it correctly orders cases (0.80 is more certain than 0.48)
but a reported "80%" is not empirically guaranteed to be right 80% of the time.
For production I'd calibrate against a labeled dataset (Platt scaling or
isotonic regression on the raw `ai_score`) and check it with a reliability
diagram, so the percentage shown to a reader is a real probability. I'd also let
each content type carry its own thresholds (poetry and academic prose behave
differently) rather than one global band.

## Transparency label — the three variants

The label is the only thing a non-technical reader sees. Each variant states the
result plainly, turns confidence into a percentage, is explicit that this is an
*estimate*, and (for AI) points to the appeal path. `{pct}` = rounded confidence.

| Variant | Verbatim text |
|---|---|
| **High-confidence AI** (`likely_ai`) | 🤖 **Likely AI-generated** — Our automated analysis estimates this content was probably created with the help of generative AI (confidence: {pct}). This is an estimate, not a certainty — AI detection is an unsolved problem. If you are the creator and believe this is wrong, you can appeal and a human will review it. |
| **High-confidence human** (`likely_human`) | ✍️ **Likely human-written** — Our automated analysis found no strong signs of AI generation in this content (confidence: {pct}). This is an estimate, not a guarantee of authorship. |
| **Uncertain** (`uncertain`) | ❓ **Attribution uncertain** — Our automated analysis could not confidently determine whether this content was written by a human or generated by AI (confidence in either direction: {pct}). Rather than guess, we are telling you honestly: treat the origin as unverified. |

Design choices: the AI label is the only one that hedges twice ("estimate, not a
certainty" *and* "unsolved problem") and is the only one that surfaces the appeal
path — because it is the only label that can harm a creator. The uncertain label
explicitly says "rather than guess… we are telling you honestly," because the
goal is to make uncertainty feel like a deliberate, trustworthy choice rather
than a system failure.

## Appeals workflow

A creator who believes they were misclassified sends:

```bash
curl -s -X POST http://127.0.0.1:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-CONTENT-ID", "creator_reasoning": "I wrote this myself from personal experience. I am a non-native English speaker and my writing style may appear more formal than typical."}'
```

The endpoint (1) looks up the original decision by `content_id` (404 if
unknown), (2) updates the content's status to `under_review`, (3) appends an
audit-log row (event `appeal`) carrying the creator's reasoning *beside* the
original verdict and scores, and (4) returns a confirmation. There is **no
automated re-classification** — a human reviewer is the decision-maker; the
appeal simply routes the case to them. A reviewer opens the queue via
`GET /log` and sees, per `event=appeal` row, the original attribution +
confidence, both signal scores, and the creator's reasoning side by side.

Real confirmation response:
```json
{
  "message": "Appeal received. This content is now under review by a human.",
  "content_id": "8cb07fa1-0e25-4274-ba8b-dc9062a7120b",
  "status": "under_review",
  "original_attribution": "uncertain",
  "original_confidence": 0.4836,
  "appeal_reasoning": "I wrote this myself from personal experience. ..."
}
```

## Rate limiting

Implemented with Flask-Limiter (in-memory storage) on `POST /submit`:

```python
@limiter.limit("10 per minute;100 per day")
```

**Chosen limits and reasoning:**

- **10 / minute** — A real creator submits their own work occasionally; even an
  active writer pasting drafts rarely exceeds a few per minute. 10/min leaves
  comfortable headroom for legitimate bursts (editing and resubmitting) while
  stopping a script from hammering the (paid, latency-bound) LLM signal. An
  adversary trying to flood the endpoint or probe the classifier hits the wall
  almost immediately.
- **100 / day** — Caps sustained abuse even if it stays under the per-minute
  limit (e.g. a slow scraper doing 9/minute all day). 100 submissions/day is far
  above any genuine single creator's needs but low enough to bound cost and
  make bulk-probing the detector impractical.

`/appeal` is separately limited to **20 / hour** — appeals are rare, human-driven
events, so a tight cap is fine and prevents appeal-spam.

**Evidence** (12 rapid requests, with 2 already used in the same minute window —
limit triggers at 10):
```
200
200
200
200
200
200
200
200
429   ← limit reached
429
429
429
```
The `429` response body:
```json
{
  "error": "rate_limit_exceeded",
  "message": "Too many submissions. Please slow down and try again shortly.",
  "detail": "10 per 1 minute"
}
```

## Known limitations

- **Formal and non-native human writing is the system's worst case.** Both
  signals key off formality: stylometry scores long words, even sentence
  lengths, and transition markers as AI-like, and the LLM independently
  over-calls academic prose. In testing, the monetary-policy paragraph (genuine
  human academic writing) reached `ai_score ≈ 0.74` — *just* below the AI
  threshold. A slightly denser formal paragraph **would** cross it and earn a
  false "likely AI" label. This is tied directly to a property of the signals,
  not a data-volume problem: neither signal can distinguish "formal because a
  machine wrote it" from "formal because the human writer is precise or
  non-native." The high AI threshold and the appeal path are the mitigations,
  not a fix.
- **Short texts and repetitive poetry** give stylometry too few sentences for
  variance to mean anything, and heavy repetition flattens lexical variety —
  both push toward a spurious "uniform → AI" reading.
- **Lightly edited AI** is designed to land in `uncertain`, which is honest but
  means the system will not catch a determined human-in-the-loop AI user.

If deploying for real, I would add per-creator calibration (a known-human
writer's baseline stylometry), a confidence-decay for very short inputs, and
human-review SLAs on the appeal queue.

## Spec reflection

- **How the spec guided me:** writing the three label variants *verbatim* in
  `planning.md §2.3` before any UI code meant the label function was a pure
  lookup — the hard thinking (what does 48% confidence *say* to a reader?) was
  already done. Likewise, fixing the threshold table in `§2.2` first kept the
  scoring code honest; I had a target to verify against rather than inventing
  ranges as I coded.
- **How the implementation diverged:** the spec initially set the AI threshold
  at `0.70`. When I ran the four calibration inputs, the formal-human paragraph
  scored `0.7377` and was labeled `likely_ai` — a false positive on a human's
  work, the exact outcome the spec said was worst. I raised the threshold to
  `0.75` (and updated `planning.md` to match), widening the uncertain band so
  that case lands in `uncertain`. The spec's *principle* (false positives are
  worst) was right; the specific number needed empirical tuning.

## AI usage

- **Stylometric signal function.** I directed the AI to implement
  `stylometric_signal` given the four sub-metrics named in `planning.md §2.1`
  (burstiness, long-word ratio, transition density, informality) and the
  required output shape (a single `[0,1]` score plus a breakdown dict). It
  produced a reasonable first pass, but its normalization constants were
  arbitrary and its sub-score weights didn't match my intent. I overrode the
  weights to `0.30/0.20/0.25/0.25` and set the normalization anchors
  (e.g. coefficient-of-variation reference `0.6`, long-word ratio `0.25`) by
  hand-running the spec's clearly-AI and clearly-human samples until the
  sub-scores landed where I expected.
- **Confidence combination + thresholds.** I asked the AI to write the
  `classify` combiner from `§2.2`. Its draft mapped `ai_score` to verdicts but
  used symmetric thresholds (0.5 either side) — which silently ignored the whole
  false-positive asymmetry the spec is built around. I rewrote the thresholds to
  be asymmetric (`0.75` / `0.35`), added the signal-disagreement penalty to the
  confidence formula, and added the disagreement-override to `uncertain`, none
  of which were in the generated version.
